## Найденные проблемы (в ожидании рефакторинга)

### Файл стилей: `myconfig.sty` (РЕШЕНО)
**Позиция Редактора/Верстальщика:**
1. [x] **Опции пакета xcolor (Критично)**: На строке 36 вызывается `\RequirePackage[table]{xcolor}`, а на строке 40 — `\RequirePackage[dvipsnames]{xcolor}`. Это вызовет ошибку компиляции "Option clash for package xcolor". Нужно объединить вызовы: `\RequirePackage[table, dvipsnames]{xcolor}`.
2. [x] **Отсутствует пакет float (Критично)**: В макросах `\smartfigure` и `\placeholderfigure` (строки 237 и 257) используется жесткое позиционирование `\begin{figure}[H]`. Для работы параметра `[H]` обязательно требуется пакет `float`. Его нет в преамбуле. Нужно добавить `\RequirePackage{float}`.
3. [x] **Цветовое выделение абзацев (Критично для печати)**: Макросы `\note`, `\warning`, `\tip` (строки 229-231) просто меняют цвет всего абзаца (`\color{RoyalBlue}` и т.д.). По ГОСТам и правилам книжной верстки это недопустимо: цветной текст теряет контрастность, а при черно-белой печати (типично для методичек) он превратится в нечитаемый блекло-серый. Текст должен оставаться черным, а выделение следует делать за счет фона (светлая плашка/рамка) с помощью пакета `tcolorbox` или отступов.

## План решения проблемы логической последовательности (Главы 1 и 2)

**Проблема:** В главе 1 описывается интерфейс PyCharm и установка библиотек в проект, но само создание проекта происходит только в начале главы 2! Пользователь физически не может открыть терминал или создать `requirements.txt`, не создав проект.

**Решение (Рефакторинг структуры):**
Необходимо перенести подраздел «Создание проекта» (нынешний 2.1) из второй главы в первую, сразу после установки PyCharm.
Новая логика Главы 1:
- 1.1 Установка Python
- 1.2 Установка PyCharm
- **1.3 Создание первого проекта** *(перенесено из 2.1)*
- 1.4 Знакомство с интерфейсом *(теперь логично, интерфейс открыт)*
- 1.5 Установка библиотек *(проект есть, терминал доступен)*
- 1.6 Проверка работоспособности

Новая логика Главы 2 (чистый фокус на коде):
- 2.1 Модульная структура проекта (архитектура)
- 2.2 Файл конфигурации (`config.py`)
- 2.3 Модуль утилит (`utils.py`)

### Оформление листингов кода (minted) (РЕШЕНО)
**Позиция Студента-троечника:**
1. [x] **Где номера строк?! (Критично)**: В книжках и статьях преподаватель всегда говорит: «посмотрите на 12-ю строку». А здесь номеров строк нет. Приходится считать их пальцем по бумаге. Это очень неудобно!

**Позиция Редактора/Верстальщика:**
1. [x] **Нарушение манифеста (Критично)**: В `manifest.md` и инструкциях к проекту строго прописано: «Every code block uses `minted` with syntax highlighting, **line numbers**, and captions». В файле `myconfig.sty` (строки 187-196) в команде `\setminted` забыт параметр `linenos=true`.
2. [x] **Проблема копирования из PDF**: Формат PDF технически неприспособлен для корректного переноса отступов и символов (часто съезжают табы, кавычки-елочки). Вместо отключения номеров строк правильной книжной и методической практикой является строгое указание не копировать код из PDF, а использовать приложенные файлы исходников (`src/`). Я вернул номера строк и добавил блок `\warning` в главу 1, который объясняет эту проблему студентам.
3. [x] **Слово "Listing" вместо "Листинг"**: Код подписан английским "Listing 1". Для русскоязычной методички это ошибка. Добавлено переопределение `\renewcommand{\listingscaption}{Листинг}`.
4. [x] **Визуальное оформление**: `frame=single` рисует жесткую черную рамку вокруг кода. В современной книжной верстке технических пособий чаще используют легкую заливку фона (например, `bgcolor=lightgray!20`) без резких рамок, либо используют связку `tcolorbox` + `minted`. Тем не менее, ручной хак с цветом комментариев (`\makeatletter ...`) выполнен превосходно — это действительно повысит читаемость при черно-белой печати.
5. [x] **Обертка блоков**: Использование `\begin{minipage}{\linewidth} \captionof...` вокруг каждого листинга — рабочий прием для защиты от разрыва на страницах, но засоряет код. На будущее лучше настроить глобальное окружение через `tcolorbox`, хотя пока это не критично.
