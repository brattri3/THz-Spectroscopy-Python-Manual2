\section{Инженерное проектирование: Конфигурация, Утилиты и СУБД-архитектура}

Прежде чем бросаться писать формулы БПФ, настоящий инженер чертит план. Хаотичный код, где всё свалено в один огромный файл, быстро становится нечитаемым. В этом разделе мы создадим проект в PyCharm, продумаем его модульную структуру, разработаем концепцию перевода физических величин, внедрим парадигму систем управления базами данных (СУБД) для хранения результатов и напишем наш первый полезный файл — конфигурационный.

\subsection{Организация файлов}

\subsubsection{Модульная структура}
Представь, что мы собираем сложный физический прибор. Мы не будем паять все детали на одной плате в виде огромного комка проводов. Мы разобьем прибор на функциональные блоки: блок питания, процессор, дисплей.

В программировании это называется \textbf{модульностью}. Мы разделим нашу программу на логические блоки (модули). Каждый модуль (отдельный \texttt{.py} файл) будет решать только свою узкую задачу.

Вот как будет выглядеть архитектура нашего «прибора»:
\begin{itemize}
    \item \texttt{data/} --- папка с исходными текстовыми файлами сигналов (наши «сырые» данные).
    \item \texttt{config.py} --- пульт управления. Здесь мы соберем все настройки и пути к файлам.
    \item \texttt{utils.py} --- модуль утилит. Перевод физических величин и нормализация геометрии.
    \item \texttt{data\_loader.py} --- блок загрузки данных. Читает и структурирует файлы под СУБД-хранилище.
    \item \texttt{spectrum.py} --- блок цифровой обработки (DSP). Удаляет постоянное смещение, накладывает Гауссово окно, вычисляет БПФ без искусственной интерполяции.
    \item \texttt{theoretical.py} --- физико-математический блок. Содержит расчет Blanco-модели и матрицы Джонса.
    \item \texttt{main.py} --- главное интегрированное приложение (GUI) и запуск всего конвейера обработки.
\end{itemize}

Обрати внимание: мы объединили логику визуализации, интерактивной подгонки и главный координационный скрипт в один файл \texttt{main.py}. Это позволяет избежать путаницы между запуском программы и интерфейсными модулями.

Теперь создай в PyCharm все эти файлы (пока пустые) и папку \texttt{data}.

\subsection{Файл конфигурации}

\subsubsection{Зачем нужен config.py}
Любой программный проект начинается с настройки базовых путей к данным. Самая частая ошибка новичка — жестко зашивать пути к файлам на своем жестком диске (например, \texttt{"C:/Users/Student/Desktop/data/..."}) прямо в код загрузчиков. Когда проект переносится на другой компьютер, такой код мгновенно ломается.

Мы поступим профессионально: создадим централизованный файл настроек \texttt{config.py}, который будет определять положение папок относительно самого исполняемого скрипта. Другие модули будут просто импортировать эти настройки.

По мере того, как мы будем развивать физическую модель и добавлять параметры фильтрации или оптимизации, мы будем расширять наш конфигурационный файл, добавляя в него новые константы. Но начнем с самого фундаментального — определения путей.

\subsubsection{Начальная разметка config.py}

Давай напишем код для файла \texttt{config.py}.

\warning{\textbf{Осторожно при копировании из PDF!} Программы для чтения PDF часто «съедают» пробелы в начале строк при копировании. Python строго требует правильных отступов (indentation). Если ты скопируешь код из методички в PyCharm и он подчеркнет его красным, тебе придется вручную восстановить отступы с помощью клавиши \texttt{Tab} или пробелов. Лучший способ научиться — \textbf{перепечатывать код вручную}, так ты быстрее запомнишь синтаксис!}

\begin{minipage}{\linewidth}
	\captionof{listing}{Создаем пульт управления: config.py}
	\begin{minted}{python}
from pathlib import Path

# --- ПУТИ К ДАННЫМ ---
# Используем pathlib для автоматического и платформонезависимого построения путей
BASE_DIR = Path(__file__).resolve().parent
DATA_DIR = BASE_DIR / "data"

# Проверим правильность путей при прямом запуске файла config.py
if __name__ == "__main__":
    print(f"Базовая директория проекта: {BASE_DIR}")
    print(f"Путь к папке с данными: {DATA_DIR}")
	\end{minted}
\end{minipage}

\textbf{Давай детально разберём, что здесь происходит и как это работает:}
\begin{itemize}
    \item \texttt{from pathlib import Path}: Модуль \texttt{pathlib} — это современный стандартный инструмент Python для работы с путями файловой системы. В отличие от старого модуля \texttt{os.path}, оперирующего сырыми строками, \texttt{pathlib} представляет пути в виде специальных умных объектов. Он автоматически определяет операционную систему и сам расставляет правильные разделители: прямые слеши (\texttt{/}) в Linux/macOS или обратные слеши (\texttt{\textbackslash}) в Windows.
    \item \texttt{\_\_file\_\_}: Это встроенная служебная переменная Python, которая автоматически создается при запуске скрипта и хранит абсолютный путь к самому файлу, в котором она написана (в данном случае — к файлу \texttt{config.py}).
    \item \texttt{Path(\_\_file\_\_).resolve().parent}: Конструкция последовательно находит полный путь к файлу с разрешением всех символических ссылок (\texttt{.resolve()}), а затем поднимается на один уровень выше к родительской папке (\texttt{.parent}). Так мы динамически вычисляем корневую директорию нашего проекта \texttt{BASE\_DIR}, где бы он ни лежал.
    \item \texttt{BASE\_DIR / "data"}: В библиотеке \texttt{pathlib} разработчики языка переопределили стандартный оператор математического деления \texttt{/}. Когда левым операндом выступает объект пути, этот символ работает как платформонезависимый конкатенатор (соединитель / склеиватель) путей. Это позволяет изящно и надежно пристыковать папку \texttt{data} к корню проекта, не беспокоясь о различиях операционных систем.
    \item \texttt{if \_\_name\_\_ == "\_\_main\_\_":}: Это стандартный паттерн входа в Python. Переменная \texttt{\_\_name\_\_} хранит имя модуля. Если мы запускаем файл \texttt{config.py} напрямую через терминал или среду разработки, Python присваивает этой переменной значение \texttt{"\_\_main\_\_"}. Если же мы импортируем этот файл в другой скрипт (например, напишем \texttt{import config} в модуле загрузки данных), то условие не выполнится. Благодаря этому отладочные сообщения не будут засорять консоль при нормальной работе всей программы, но позволят быстро протестировать пути при изолированном запуске конфига.
    \item \texttt{print()}: Встроенная функция вывода информации. Она принимает переданные ей аргументы, преобразует их в текстовый вид и выводит на экран (в командную строку).
    \item \texttt{f"..."} (f-строки или форматированные строковые литералы): Современный и наиболее удобный способ форматирования строк в Python (появился в версии 3.6). Префикс \texttt{f} перед открывающей кавычкой указывает интерпретатору, что строка является динамическим шаблоном. Любые выражения или переменные, помещенные внутри фигурных скобок \texttt{\{\}}, будут вычислены в реальном времени и автоматически подставлены в результирующий текст в виде строк.
\end{itemize}

Запустить файл \texttt{config.py} для проверки путей можно двумя способами:
\begin{enumerate}
    \item \textbf{Через графический интерфейс PyCharm:} Нажми зелёный треугольник («Run» / запуск) в правом верхнем углу экрана или кликни правой кнопкой мыши в любом месте внутри кода файла \texttt{config.py} и выбери пункт \textbf{Run 'config'}. Это самый простой и наглядный способ для новичка.
    \item \textbf{Через встроенный терминал PyCharm:} Открой вкладку Terminal в нижней части экрана и введи команду запуска.
\end{enumerate}

При запуске через командную строку вводи саму команду без служебных символов приглашения (таких как знак доллара \texttt{\$} или процента \texttt{\%}, которые часто приводятся в учебниках как обозначение начала строки ввода):
\begin{minted}{text}
python config.py
\end{minted}

В обоих случаях в панели запуска или консоли ты увидишь примерно следующий вывод (конкретные пути будут зависеть от того, в какой именно папке на диске сохранен проект):
\begin{minted}{text}
Базовая директория проекта: C:\THz_Project
Путь к папке с данными: C:\THz_Project\data
\end{minted}

Убедившись, что пути выводятся абсолютно правильно, мы можем быть уверены, что наша конфигурационная база готова. Теперь другие части программы смогут ссылаться на \texttt{config.DATA\_DIR}, не задумываясь о том, на каком компьютере или сервере запущена программа.

\warning{Никогда не используй абсолютные пути вроде \texttt{C:\textbackslash Users\textbackslash ...\textbackslash data}. Если ты перенесешь проект на другой компьютер, путь сломается. Конструкция \texttt{BASE\_DIR / "data"} делает путь \textbf{относительным} — он всегда указывает на папку \texttt{data} внутри текущего проекта.}

\subsection{Разделение уровней абстракции: Модуль утилит utils.py}

Для поддержания чистоты кода и разделения ответственности (Separation of Concerns, принципа разделения обязанностей модулей) мы создаём модуль вспомогательных функций \texttt{utils.py}. Физическое ядро программы должно оперировать строго фундаментальными единицами СИ (метры, секунды, Герцы, линейные амплитуды). Однако пользователю и лаборанту намного привычнее видеть на ползунках микрометры, пикосесунды, децибелы и градусы. 

Модуль \texttt{utils.py} выступает «мостом», изолирующим математику от презентационного слоя (интерфейса пользователя). Прежде чем писать код для загрузки и обработки сложных данных, давай создадим этот модуль и напишем нашу первую вычислительную функцию. На её примере мы подробно разберем базовый синтаксис (правила написания кода) функций, математические операторы и условные конструкции в Python.

\subsection{Пишем свою первую вычислительную функцию: Нормализация углов}

\subsubsection{Постановка физической задачи}

В установках ТГц-спектроскопии ротатор (подвижный держатель оптического элемента) первого поляризатора может иметь шкалу углов (лимб) от $0^\circ$ до $360^\circ$. Однако физические законы взаимодействия излучения с решетками поляризаторов симметричны. Кроме того, теоретическая модель Blanco для расчета пропускания требует, чтобы углы ориентации поляризатора лежали в симметричном интервале $[-180^\circ, 180^\circ]$ относительно оптической оси пропускания.

Это означает, что углы больше $180^\circ$ эквивалентны отрицательным углам. Например, поворот на $280^\circ$ физически тождественен повороту в обратную сторону на угол $280^\circ - 360^\circ = -80^\circ$. Нам нужна функция, которая приведет любой экспериментальный угол к физически корректному интервалу $[-180^\circ, 180^\circ]$.

Мы можем решить эту задачу двумя разными логическими способами. Давай рассмотрим оба, чтобы ты мог выбрать наиболее понятный.

\mybold{Способ 1: Логическое ветвление через \texttt{if}}
\begin{enumerate}
    \item Сначала ограничим любой входной угол рамками одного полного оборота $[0, 360^\circ)$ с помощью взятия остатка от деления: \texttt{angle = angle \% 360}. Это уберет лишние обороты, если угол, например, равен $730^\circ$.
    \item Затем проверим: если получившийся угол больше $180^\circ$, мы вычитаем из него $360^\circ$ (производим декремент — уменьшение значения переменной), переводя его в отрицательную область. Иначе оставляем как есть.
\end{enumerate}

\mybold{Способ 2: Чистая математическая формула}
Существует изящный способ сделать это без ветвлений в одну строку с помощью формулы:
\begin{equation}
\text{Normalized Angle} = (\text{Angle} + 180 \pmod{360}) - 180
\end{equation}
В коде Python это записывается чрезвычайно лаконично: \texttt{((angle + 180) \% 360) - 180}. Давай проверим на примере угла $280^\circ$:
\begin{equation}
(280 + 180) \pmod{360} - 180 = 460 \pmod{360} - 180 = 100 - 180 = -80^\circ
\end{equation}
Формула работает безупречно и очень быстро, так как выполняется за один такт процессора без ветвлений.

\subsubsection{Реализация модуля утилит utils.py}

Давай создадим в корне нашего проекта файл \texttt{utils.py} и запишем туда оба варианта нашей вычислительной функции. В основном коде мы будем использовать лаконичный математический вариант, а первый оставим в качестве наглядного комментария для самопроверки.

\begin{minipage}{\linewidth}
	\captionof{listing}{Вспомогательные функции в файле utils.py}
	\label{lst:utils_code}
	\begin{minted}[linenos, numbersep=5pt, frame=lines, framesep=2mm]{python}
def normalize_angle_conditional(angle: float) -> float:
    """
    Приводит произвольный угол к симметричному диапазону [-180, 180] градусов
    с использованием классического логического ветвления (if/else).
    """
    # 1. Ограничиваем угол диапазоном [0, 360) градусов
    angle = angle % 360.0
    
    # 2. Если угол зашел во вторую половину круга, переводим в отрицательную шкалу
    if angle > 180.0:
        angle -= 360.0  # тождественно: angle = angle - 360.0
        
    return angle


def normalize_angle(angle: float) -> float:
    """
    Приводит произвольный угол к симметричному диапазону [-180, 180] градусов
    с использованием компактной арифметической формулы с остатком от деления.
    """
    return ((angle + 180.0) % 360.0) - 180.0


if __name__ == "__main__":
    # Быстрый тест работоспособности функций
    test_angles = [0.0, 90.0, 180.0, 270.0, 360.0, 730.0, -45.0]
    print("Проверка математической нормализации углов:")
    for a in test_angles:
        norm_cond = normalize_angle_conditional(a)
        norm_math = normalize_angle(a)
        print(f"Исходный: {a:>6.1f}° | Ветвление: {norm_cond:>6.1f}° | Формула: {norm_math:>6.1f}°")
\end{minted}
\end{minipage}

Исходный:   90.0° | Ветвление:   90.0° | Формула:   90.0°
Исходный:  180.0° | Ветвление:  180.0° | Формула:  180.0°
Исходный:  270.0° | Ветвление:  -90.0° | Формула:  -90.0°
Исходный:  360.0° | Ветвление:    0.0° | Формула:    0.0°
Исходный:  730.0° | Ветвление:   10.0° | Формула:   10.0°

Однако компактная формула \texttt{normalize\_angle} предпочтительнее в научных расчетах, поскольку она работает без инструкций ветвления процессора (conditional jumps, условных переходов на уровне ассемблерных команд), что ускоряет выполнение при массовой векторной обработке больших массивов данных.

\subsubsection{Разбор кода: Анатомия функции и операторы}

Давай разберем основные синтаксические элементы создания функций на языке Python, использованные в нашем коде. Функция в программировании — это изолированный блок кода, который принимает входные данные (аргументы), выполняет с ними определенные действия и возвращает результат. Её можно сравнить с математической формулой $y = f(x)$ или «чёрным ящиком».

\mybold{Анатомия функции в Python}
\begin{itemize}
    \item \mybold{Ключевое слово \texttt{def}:} Сигнализирует интерпретатору (программе выполнения кода) о начале объявления функции.
    \item \mybold{Имя функции (идентификатор):} Записывается строчными буквами, слова разделяются нижним подчеркиванием (\texttt{snake\_case} — змеиный стиль, принятый в Python). Имя должно четко отражать суть выполняемого действия.
    \item \mybold{Список параметров (аргументов):} Перечисляется в круглых скобках после имени функции. Это локальные переменные, в которые будут подставлены реальные значения при вызове.
    \item \mybold{Двоеточие (\texttt{:}):} Ставится в конце строки объявления. Оно сообщает Python, что далее следует тело функции.
    \item \mybold{Отступы (indentation):} В отличие от многих языков, где блоки кода выделяются фигурными скобками \texttt{\{\}}, в Python блоки выделяются строго отступами (стандарт — ровно 4 пробела). Весь код, имеющий одинаковый сдвиг вправо, принадлежит одному блоку.
    \item \mybold{Документирование (\texttt{docstring}):} Строка текста, заключенная в тройные кавычки \texttt{"""} сразу после заголовка функции. Она описывает назначение функции, аргументы и возвращаемое значение. Это признак профессионального стиля кода.
    \item \mybold{Ключевое слово \texttt{return}:} Завершает выполнение функции и возвращает вычисленный результат (выходной параметр) вызывающему коду. Если его опустить, функция вернет специальный пустой объект \texttt{None}.
\end{itemize}

\note{Обрати внимание на двоеточия после типов аргументов \texttt{angle: float} и стрелочку \texttt{-> float}. Это аннотации типов (type hints, подсказок типов) в Python. Они не влияют на скорость работы программы, но помогают современным средам разработки вроде PyCharm мгновенно подсказывать ошибки, если ты случайно попытаешься передать в функцию текстовую строку вместо числа.}

\mybold{Математические операторы и логическое ветвление}

Для выполнения расчетов Python предлагает стандартный набор арифметических операций:
\begin{itemize}
    \item \texttt{+} и \texttt{-} --- сложение и вычитание;
    \item \texttt{*} и \texttt{/} --- умножение и деление (деление всегда возвращает число с плавающей точкой);
    \item \texttt{\%} --- остаток от деления (взятие по модулю). Например, \texttt{10 \% 3} вернет \texttt{1}, так как $10 = 3 \times 3 + 1$.
\end{itemize}

Логическое ветвление (условный переход для выполнения разных ветвей алгоритма в зависимости от данных) реализуется с помощью конструкции \texttt{if}, \texttt{elif} (сокращение от else if — «иначе если») и \texttt{else} («иначе»). Каждое условие завершается двоеточием, а код внутри ветвей сдвигается еще на 4 пробела вправо.

Полный листинг модуля \texttt{utils.py} приведен в \textbf{Приложении \ref{app:utils}}.

\subsection{СУБД-парадигма: Единое хранилище данных проекта \texttt{data\_store}}

В типовых студенческих скриптах обработка данных часто превращается в хаос: создаются десятки независимых переменных вида \texttt{signals\_raw}, \texttt{background\_signals\_raw}, \texttt{spectra}, \texttt{transmission\_curves}... При этом связь между файлом сигнала образца и соответствующим ему референсным фоном (background) теряется или жестко зашивается через индексы в массивах, что приводит к критическим ошибкам при изменении состава эксперимента.

Чтобы решить эту проблему раз и навсегда, мы применим профессиональный архитектурный паттерн — \textbf{внутрипамятьевую реляционную базу данных}. Вся информация в нашей программе будет жить в едином централизованном словаре \texttt{data\_store}.

\subsubsection{Схема составных ключей}

Ключом в словаре \texttt{data\_store} является строго типизированный кортеж (tuple, неизменяемый упорядоченный список) из четырёх элементов:
\begin{equation}
\text{Key} = (\text{Tag}, \text{Angle}_1, \text{Angle}_2, \text{Repetition})
\end{equation}

Где каждый элемент имеет чёткий физический и логический смысл:
\begin{enumerate}
    \item \textbf{\texttt{Tag} (строка):} Уникальный идентификатор типа данных:
    \begin{itemize}
        \item \texttt{'signal\_raw'} --- сырой временной ТГц-сигнал образца для конкретной реплики (прохода).
        \item \texttt{'bg\_raw'} --- сырой временной референсный сигнал (фон) для конкретного прохода.
        \item \texttt{'spec\_avg'} --- итоговый спектр пропускания по мощности исследуемого образца, полученный путем поштучного вычисления спектра пропускания каждого прохода (отношение мощности сигнала к мощности парного фона) с последующим частотным усреднением по всем повторениям.
        \item \texttt{'spec\_bg\_avg'} --- усредненный спектр мощности опорного сигнала (референса), используемый для контроля динамического диапазона установки.
    \end{itemize}
    \note{Предложенный набор тегов является открытым и расширяемым. Проектируемая архитектура базы данных полностью гибкая: студент может вводить любые собственные кастомные теги (например, \texttt{'signal\_averaged\_time'} для временного усреднения, \texttt{'phase\_spectrum'} для фазового спектра или любые промежуточные метки) для фильтрации, кэширования и хранения расчетных данных.}
    \item \textbf{\texttt{Angle1} (число с плавающей точкой):} Нормализованный угол установки первого поляризатора (ротатора, подвижного держателя оптического элемента) в градусах.
    \item \textbf{\texttt{Angle2} (число с плавающей точкой):} Угол установки второго поляризатора (анализатора).
    \item \textbf{\texttt{Repetition} (целое число или \texttt{None}):} Номер повторного измерения (прохода). Для обработанных, усредненных или спектральных данных, где реплики уже объединены, этот параметр равен \texttt{None}.
\end{enumerate}

\warning{\textbf{Критически важное физическое замечание!}
Студенты часто пытаются сначала усреднить временные ТГц-импульсы (time-domain signals), полученные в нескольких проходах, а затем рассчитать БПФ. \textbf{Делать это категорически запрещено!} 
Из-за температурного расширения элементов, механического люфта линии задержки и микровибраций оптического стола во время эксперимента возникает временной дрейф (так называемый «джиттер» (jitter, случайное дрожание или временной сдвиг фазы импульса)). Это означает, что ТГц-импульсы в разных проходах могут быть слегка смещены по времени друг относительно друга на фемтосекундные масштабы. 
Если вы усредните их во временной области, небольшие фазовые сдвиги приведут к деструктивной интерференции (взаимному гашению встречных фаз волн) на высоких частотах. Спектр усредненного сигнала будет содержать искусственное затухание в области 1-2 ТГц. 
Физически корректный подход: рассчитать комплексные спектры для каждого повторения отдельно, взять их модуль (абсолютное значение комплексного числа, соответствующее амплитудному спектру) и усреднять уже \textbf{в частотной области}. Наша структура ключей \texttt{data\_store} прямо форсирует эту логику, храня отдельные сырые сигналы и спектры проходов под соответствующими номерами \texttt{Repetition}.}

\subsubsection{Архитектурные преимущества СУБД-подхода}

\begin{itemize}
    \item \textbf{Абсолютная связность:} При загрузке файлов каждый замер образца автоматически привязывается к своему индивидуальному фону через метаданные угла и номера повторения.
    \item \textbf{Унификация программного интерфейса (API):} Все расчетные функции и графические интерфейсы программы больше не требуют передачи десяти аргументов. Они принимают один единственный объект — \texttt{data\_store}, производят с ним манипуляции и возвращают его же, обогащенного новыми тегами.
    \item \textbf{Элегантная фильтрация (SQL-like):} Сделать выборку нужных данных становится чрезвычайно просто с помощью встроенных в Python генераторов словарей (dict comprehensions, компактного синтаксиса сборки новых словарей «на лету»).
\end{itemize}

Например, извлечь из базы данных все усредненные спектры образцов для построения графиков можно одной лаконичной строчкой:
\begin{minted}{python}
# Фильтрация словаря (dict comprehension) по первому элементу ключа:
spectra_to_plot = {
    k: v                         # 5. Собираем в новый словарь пары k (ключ) и v (данные)
    for k, v in data_store.items()  # 1. Перебираем пары "ключ-значение" из исходного хранилища
    if k[0] == 'spec_avg'        # 3. Фильтр: берем только те ключи, у которых Tag равен 'spec_avg'
}

# Подробный разбор механики:
# 1. `data_store.items()` возвращает итератор (специальный объект для последовательного перебора элементов в цикле) пар (ключ, значение).
# 2. Ключ `k` представляет собой кортеж (Tag, Angle1, Angle2, Repetition).
# 3. `k[0]` извлекает самый первый элемент этого кортежа — строку-тег (Tag).
# 4. Условие `if k[0] == 'spec_avg'` отсекает все сырые сигналы и оставляет только спектры.
# 5. Выражение `{k: v ...}` конструирует новый отфильтрованный словарь.
\end{minted}

Это делает код чрезвычайно читаемым, легко расширяемым и полностью защищенным от случайной перепутанности данных. Чтобы закрепить этот важный архитектурный навык, реши следующее мини-задание.
